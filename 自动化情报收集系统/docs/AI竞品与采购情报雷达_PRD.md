## 1. 产品背景

AI 产品信息高度分散在政策网站、研究报告、竞品官网、招聘平台、招标采购平台、行业会议、企业年报、社交媒体、合作伙伴公告和技术社区中。

仅仅收集新闻会产生三个问题：

1. 信息数量持续增长，但无法形成可复用判断；
2. 竞品发布、企业招聘和客户采购之间缺少关联；
3. 日常研究结果无法直接服务于产品规划、商业判断、客户机会和面试表达。

本产品将外部信息转化为结构化的竞品与采购情报，并每天自动沉淀到本地文件和 Notion 知识库。

## 2. 产品定位

### 2.1 核心定位

面向企业 AI 平台产品经理、AI Business Product、企业 AI 解决方案负责人和商业分析人员的轻量情报雷达。

系统重点回答：

- 企业为什么采购 AI；
- 客户正在建设什么能力；
- 预算出现在哪些行业和业务场景；
- 竞品新增了什么功能、价格和商业模式；
- AI 如何进入 CRM、ERP、OA、ITSM、知识库和业务流程；
- 哪些外部信号值得转化为产品需求、销售线索或职业案例。

### 2.2 非目标

当前版本不重点解决：

- 大模型训练、微调和评测平台；
- GPU 调度、CUDA、底层推理集群监控；
- 全网无差别新闻聚合；
- 付费数据平台的账号自动化和反爬绕过；
- 自动替代人工完成最终采购决策。

## 3. 目标用户

### 3.1 核心用户

| 用户 | 核心诉求 |
| --- | --- |
| 企业 AI 产品经理 | 发现竞品能力、产品趋势和业务落地方式 |
| AI 解决方案负责人 | 识别客户行业、采购预算、交付标准和合作机会 |
| 商业化产品经理 | 研究价格、计费、销售方式、渠道和 ROI |
| 求职与面试用户 | 沉淀有证据的行业判断、竞品案例和产品表达 |

### 3.2 用户任务

1. 每天快速查看高价值变化；
2. 按竞品查看功能、价格、客户和生态动态；
3. 识别采购意向、招标、中标、合同与预算；
4. 将重要信号写入 Notion；
5. 每周形成趋势判断、产品机会和面试素材。

## 4. 产品目标与衡量指标

### 4.1 产品目标

- 建立稳定的 AI 竞品和采购信息采集管道；
- 将信息自动归类到十类情报信号；
- 固定追踪 10 个与企业 AI 落地相关的重点竞品；
- 自动识别采购阶段、预算、厂商和证据等级；
- 每天自动生成报告并可推送到 Notion；
- 降低人工浏览、复制、整理和重复判断成本。

### 4.2 核心指标

| 指标 | 定义 | 当前目标 |
| --- | --- | --- |
| 任务成功率 | 定时任务完整执行次数 / 应执行次数 | ≥ 95% |
| 数据源成功率 | 成功采集数据源 / 启用数据源 | ≥ 80% |
| 竞品覆盖率 | 当日有有效信号的重点竞品 / 10 | 日报展示，不强制满覆盖 |
| 高价值信号率 | 中高可信度信号 / 有效信号 | ≥ 30% |
| 明确采购信号数 | 包含 intent、tender、award 或 partnership 的数量 | 持续观察 |
| Notion 同步成功率 | 成功创建页面 / 发起同步次数 | ≥ 95% |
| 人工复盘转化率 | 进入需求池、机会池或面试素材的信号 / 高价值信号 | 每周复盘 |

## 6. 核心业务流程

```text
数据源配置
  ↓
定时触发 / 手动运行
  ↓
RSS、RSSHub、官方域名搜索源采集
  ↓
标准化标题、摘要、链接、时间和来源
  ↓
关键词过滤与竞品别名归属
  ↓
十类信号分类
  ↓
采购阶段、预算、厂商和证据识别
  ↓
评分、可信度和去重
  ↓
JSONL / Summary / Markdown 日报
  ↓
可选推送 Notion
  ↓
周复盘：需求池、机会池、竞品矩阵、面试素材
```

## 7. 功能需求

### 7.1 FR-01 数据源配置

系统应支持：

- `feed_url`：原生 RSS 或 Atom；
- `rsshub_path`：通过 RSSHub 转换的数据源；
- 官方域名 Google News RSS 搜索；
- 数据源优先级、稳定性、语言和证据等级；
- 单源条数限制和最大数据源数量。

验收标准：

- 单个数据源失败不会中断全部任务；
- 失败信息输出到日志；
- 可通过 JSON 配置新增来源而不修改核心逻辑。

### 7.2 FR-02 十类信号分类

系统应根据数据源标签和关键词，将信息归入十类信号之一。

验收标准：

- 每条有效信号存在 `signal_category` 和中文 `category_label`；
- 分类关键词可配置；
- 无法归类的信息不得伪造确定结论。

### 7.3 FR-03 竞品清单管理

每个竞品应配置：

- 唯一 ID；
- 厂商、产品、区域和产品层次；
- 定位和个人相关性；
- 产品别名和监控关键词；
- 官网、价格页和独立动态源；
- 重点观察维度。

验收标准：

- 系统启动时校验竞品数量为 10；
- ID 必须唯一；
- 必须存在厂商、产品、官网和采集源；
- 支持通过别名将信息归属到具体竞品。

### 7.4 FR-04 竞品动态归属

系统应优先使用数据源绑定的 `competitor_id`，否则通过最长别名匹配归属竞品。

输出字段：

- `competitor_id`；
- `competitor_vendor`；
- `competitor_product`；
- `competitor_relevance`；
- `competitor_dimensions`。

验收标准：

- 竞品独立数据源采集的信息必须能够稳定归属；
- 通用数据源信息命中明确别名时能够归属；
- 无明确归属时保持为空，不强行匹配。

### 7.5 FR-05 采购阶段识别

支持识别：

| 阶段 | 典型关键词 |
| --- | --- |
| `intent` | 采购意向、预算公开、采购计划 |
| `tender` | 招标、询价、竞争性磋商、RFP |
| `award` | 中标、成交、合同公告 |
| `partnership` | 战略合作、生态合作、伙伴计划 |
| `unknown` | 暂无明确采购阶段 |

验收标准：出现配置关键词时输出对应阶段，否则输出 `unknown`。

### 7.6 FR-06 预算识别

系统应从文本中提取人民币和美元金额：

- 元、万元、亿元；
- USD、US$、million、billion；
- 将金额统一转换为 `budget_cny` 用于排序。

验收标准：

- 保留原始金额表达 `budget_text`；
- 复杂表格、PDF 和汇率必须标记为需要人工复核；
- 不将模型 Token 单价误判为项目总预算时，应在后续版本优化语境判断。

### 7.7 FR-07 情报评分

评分考虑：

- 类别权重；
- 证据等级；
- 是否存在明确采购阶段；
- 是否存在预算；
- 预算规模；
- 是否命中竞品或供应商；
- 关键词密度。

输出：

- `score`：0–10；
- `confidence`：high、medium、low；
- `recommended_action`：建议动作。

### 7.8 FR-08 去重

去重键由标准化标题、信号类别和竞品 ID 组成。

验收标准：同一竞品、同一类别、相同标题不重复输出；不同竞品引用相同主题时允许分别保留。

### 7.9 FR-09 日报生成

日报应包含：

1. 今日有效信号、高可信信号、明确预算和采购阶段数量；
2. 重点竞品覆盖率；
3. 10 个竞品逐一展示今日信号数量、定位、个人相关性和观察重点；
4. 十类信号覆盖；
5. Top 情报的来源、预算、可信度、链接和建议动作；
6. 数据质量原则。

即使竞品当天无信号，也应显示 `0 条`，用于检查数据源健康度。

### 7.10 FR-10 输出文件

每日输出：

```text
data/competitive_procurement/signals_YYYY-MM-DD.jsonl
data/competitive_procurement/daily_YYYY-MM-DD.md
data/competitive_procurement/summary_YYYY-MM-DD.json
```

Summary 应包含：

- 总信号数和高可信信号数；
- 有预算和采购阶段的信号数；
- 竞品总数、有信号竞品数和覆盖率；
- 每家竞品的信号数和观察重点；
- Top 信号；
- Notion 同步结果。

### 7.11 FR-11 Notion 同步

系统应支持将 Markdown 日报创建为 Notion 父页面下的子页面。

配置：

```dotenv
NOTION_TOKEN=
NOTION_COMPETITIVE_PARENT_PAGE_ID=
NOTION_PARENT_PAGE_ID=
```

规则：

- 优先使用 `NOTION_COMPETITIVE_PARENT_PAGE_ID`；
- 未配置时回退到 `NOTION_PARENT_PAGE_ID`；
- 未配置 Token 时仍生成本地文件，并在 Summary 记录失败原因；
- Notion 不写入全部低价值原始信息。

### 7.12 FR-12 定时任务

GitHub Actions：

- 每天北京时间 08:30 执行；
- 支持手动触发；
- 执行单元测试；
- 运行通用情报和竞品采购雷达；
- Artifact 保留 30 天；
- 有 Secret 时同步 Notion。

macOS：

- `launchd` 每天 09:30 调用 `scripts/run_daily.sh`；
- 本地电脑关机时不会执行，因此云端任务作为主要保障。

## 9. 非功能需求

### 9.1 稳定性

- 单源失败不影响其他来源；
- 网络请求设置超时；
- 定时任务最长运行时间受控；
- 结果和日志可追溯。

### 9.2 可维护性

- 采集源、关键词、竞品清单与代码分离；
- 新增竞品优先修改 JSON；
- 核心识别函数具备单元测试；
- 文档、版本记录与代码同步更新。

### 9.3 安全性

- Notion Token 等密钥仅保存在本地 `.env` 或 GitHub Secrets；
- 不把密钥提交到仓库；
- GitHub Actions 权限保持最小化；
- 不绕过登录、付费墙或平台访问控制。

### 9.4 数据质量

- 官方公告、招标、中标、合同和年报优先；
- 媒体、招聘、社媒和搜索趋势属于弱信号；
- 同一结论至少由两个独立来源交叉验证；
- 系统输出的是决策输入，不是最终事实结论。

## 11. 版本路线图

### v1.2.0：变化检测

- 保存竞品官网和价格页历史快照；
- 自动识别新增、删除、涨价、降价和套餐变化；
- 增加数据源健康检查和失败告警。

### v1.3.0：机会与客户账户

- 增加目标客户名单；
- 聚合年报、采购、招聘和合作信号；
- 建立线索、验证、跟进、投标和复盘漏斗。

### v1.4.0：文档和附件理解

- 解析招标和年报 PDF；
- 提取评分标准、资格条件、交付周期和付款方式；
- 引入 LLM 生成结构化摘要，但保留原文证据。

### v2.0.0：经营型情报平台

- 形成竞品功能价格矩阵；
- 输出行业预算、供应商和项目密度趋势；
- 将高价值信号自动转为产品需求、解决方案假设和面试案例；
- 增加仪表盘、周报、月报和人工反馈闭环。
