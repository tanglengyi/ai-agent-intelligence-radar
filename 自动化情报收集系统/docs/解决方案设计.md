# 解决方案设计

## 需求分析

### 要解决的问题

你需要的不是“更努力刷信息”，而是一套稳定的信息雷达：

- 能更早看到官方、平台和行业变化。
- 能判断哪些信息只是新闻，哪些会改变产品方向。
- 能把信息转成求职、面试、竞品分析、作品集材料。
- 能长期低成本运行，不制造焦虑。

### 核心用户故事

- 作为 AI 产品经理求职者，我希望每天只花 20 分钟看摘要，以便保持行业敏感度。
- 作为产品经理，我希望每条信息都有来源、证据等级和影响对象，以便避免被二手解读误导。
- 作为面试准备者，我希望每周把行业变化转成产品判断，以便在面试中表达“我看趋势的方法”。
- 作为作品集建设者，我希望每月产出一个主题复盘，以便形成可展示的产品洞察。

## 功能模块

### 1. 信息源管理

当前用 `config/sources.json` 管理：

- 来源名称
- 频道
- 证据等级
- RSS 地址或网页地址
- 监控关键词

后续可以加入：

- 权重
- 抓取频率
- 是否启用
- 失败重试策略
- 账号或 Cookie 配置

### 2. 自动抓取

当前策略：

- 有 RSS/Atom：解析条目标题、链接、摘要、发布时间。
- 没有 RSS：抓网页标题、描述和关键词命中。
- 抓取失败：记录 warning，不中断整体流程。

后续策略：

- 用 RSSHub 扩展更多不可直接订阅的网站。
- 对更新日志、帮助中心、定价页做内容 diff。
- 对招聘页做岗位数量和关键词趋势追踪。

### 3. 去重与存储

当前策略：

- 用链接生成哈希 ID。
- 存储到 SQLite。
- 重复链接自动忽略。

优势：

- 本地可控。
- 不依赖 Notion API。
- 后续容易迁移到 PostgreSQL 或向量数据库。

### 4. 标签与判断

当前是规则标签：

- 类型：政策、产品、融资、技术、风险、竞品。
- 方向：Agent、算力、API、知识库、安全、监管。
- 影响对象：C端、B端、大厂、创业公司、产品经理。
- 影响判断：利好、利空、不确定。

后续可以升级为 LLM 判断：

- 发生了什么？
- 为什么重要？
- 对我做产品/面试/判断公司有什么影响？
- 是否应该进入需求池？
- 是否存在二次验证需求？

### 5. 输出层

当前输出：

- Notion CSV：`exports/notion_import.csv`
- 日报：`reports/YYYY-MM-DD-日报.md`
- 周报：`reports/YYYY-MM-DD-周报.md`

后续输出：

- 面试话术卡片
- 竞品分析表
- 月度文章大纲
- 产品机会 Radar
- 风险与合规清单

## 自动化部署方案

### 轻量方案

每天手动执行：

```bash
python3 scripts/intel_radar.py run
```

### macOS 定时方案

用 `launchd` 每天早上运行 `scripts/run_daily.sh`，每周日运行 `scripts/run_weekly.sh`。

已提供模板：`templates/com.local.ai-intel-radar.daily.plist`

安装方式见 `docs/执行节奏.md`。

### 进阶方案

RSS/网页监控 -> n8n/Make/Zapier -> LLM 摘要 -> Notion API -> 周报自动生成。

当前本地脚本相当于这个进阶方案的最小可运行内核。

## 需求池转化规则

信息进入需求池需要满足至少一个条件：

- 官方或公司公告明确改变规则、能力、价格、入口。
- 同一方向出现 2 个以上独立信号。
- 对 B 端产品能力有明确传导。
- 能形成面试表达或作品集选题。
- 能变成可执行的产品功能：权限、审计、监控、网关、知识库、工作流、人工接管。

## 风险与约束

- 很多网站没有稳定 RSS，需要 RSSHub 或页面监控补齐。
- X/微信群/即刻等社交信息需要谨慎处理，证据等级应低于官方原文。
- 自动摘要不能代替原文判断，重要信息必须回看链接。
- 招聘页可能反爬，先把它当作低频人工复盘来源。
