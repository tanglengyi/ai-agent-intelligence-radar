## 2. 已记录问题

### OBS-20260723-001：初次合并后没有线上日报和运行日志

**发生时间**

2026-07-23。

**现象**

- `main` 已包含每日 GitHub Actions 工作流；
- 查询不到实际 Workflow Run；
- 没有可下载的日报 Artifact；
- 仓库中也没有独立的运行日志和故障排查记录。

**影响范围**

- 无法在线确认日报是否成功生成；
- 无法区分工作流未执行、执行失败和数据源无结果；
- Notion 推送状态没有统一汇总入口。

**根因**

v1.1.0 在当天北京时间 08:30 的定时点之后合并到 `main`，因此当天没有自动调度记录。同时，原工作流只负责运行程序和上传业务目录，没有生成统一运行摘要和逐步骤日志。

**临时处理**

通过 GitHub API 检查合并状态和 Workflow Run，确认不是已执行但日志丢失，而是尚未触发运行。

**永久修复**

v1.2.0 增加：

- `scripts/run_daily_ops.py` 运维执行器；
- 单元测试、通用 Pipeline 和竞品雷达的逐步骤日志；
- `run_summary.json` 和 `run_summary.md`；
- GitHub Actions Job Summary；
- 零采集、低覆盖率和 Notion 失败的自动诊断；
- Artifact 增加 `data/run_logs/`；
- 《运行与故障排查手册》和本记录。

**验证方式**

合并 v1.2.0 后手动触发 `Daily AI Intelligence Radar`，确认：

1. Actions 页面出现运行摘要；
2. Artifact 包含 `data/run_logs/`；
3. 三个步骤分别有 `.log`；
4. `latest_run.json` 和 `latest_run.md` 可读取；
5. 任务失败时 Artifact 仍然上传。

**当前状态**

代码和运维结构已实现，待 v1.2.0 合并后进行首次线上手动运行验证。

## 4. 周期性复盘

每周复盘以下指标：

- 定时任务成功率；
- 平均执行耗时；
- 通用原始资讯数量；
- 日报事件数量；
- 竞品信号数量；
- 10 个重点竞品覆盖率；
- Notion 同步成功率；
- 失败数据源数量和连续失败天数；
- S1-S4 问题数量及关闭周期。

连续性规则：

- 同一源连续失败 3 天：进入数据源维护；
- 竞品覆盖率连续 3 天低于 30%：检查竞品源和关键词；
- Notion 连续失败 2 次：检查 Secret 和授权；
- 连续 2 天没有日报：按 S2 处理；
- 日报和日志同时缺失：按 S1/S2 边界处理。
