# AI 情报系统运行与故障排查手册

- 文档版本：v1.2.0
- 更新时间：2026-07-23
- 适用范围：通用 AI/Agent 情报 Pipeline、竞品与采购情报雷达、Notion 同步、GitHub Actions 和 macOS `launchd`

## 1. 运维目标

运维体系需要回答五个问题：

1. 今天任务有没有运行；
2. 哪个步骤成功或失败；
3. 生成了多少条日报和竞品信号；
4. Notion 是否同步成功；
5. 失败后应该查看哪个日志、采取什么动作。

系统采用以下运维链路：

```text
GitHub Actions / launchd
  -> scripts/run_daily_ops.py
  -> 单元测试
  -> 通用情报 Pipeline
  -> 竞品采购雷达
  -> 读取两个结构化 Summary
  -> 生成运行摘要与诊断结果
  -> GitHub Job Summary
  -> Artifact 保存日报、JSON 和逐步骤日志
```

## 2. 线上查看位置

### 2.1 GitHub Actions 运行状态

进入仓库：

```text
Actions -> Daily AI Intelligence Radar
```

每次运行可以看到：

- `success`：步骤执行成功且没有诊断警告；
- `warning`：任务完成，但存在零采集、低竞品覆盖等质量问题；
- `failed`：测试、Pipeline、竞品雷达或 Notion 出现明确错误。

任务详情页的 Summary 会显示：

- Run ID、触发方式、Commit 和总耗时；
- 单元测试、通用 Pipeline、竞品雷达的状态；
- 原始资讯数、日报事件数、竞品信号数和高可信信号数；
- 10 个竞品的当日覆盖数量；
- 通用 Notion 和竞品 Notion 的同步状态；
- 自动诊断代码和处理建议。

### 2.2 Artifact 下载

每次 Actions 运行都会生成：

```text
ai-intelligence-<github_run_id>
```

保留时间：30 天。

Artifact 包含：

```text
data/reports/
data/competitive_procurement/
data/run_logs/
```

即使主任务失败，Artifact 上传步骤也会通过 `if: always()` 执行。

## 3. 运维目录结构

```text
data/run_logs/
  latest_run.json
  latest_run.md
  YYYY-MM-DD/
    <run_id>/
      01_unit-tests.log
      02_general-pipeline.log
      03_competitive-radar.log
      run_summary.json
      run_summary.md
```

说明：

- `latest_run.json`：最近一次运行的机器可读摘要；
- `latest_run.md`：最近一次运行的人类可读摘要；
- `01_unit-tests.log`：全部单元测试 stdout、stderr 和退出码；
- `02_general-pipeline.log`：通用情报 Pipeline 完整日志；
- `03_competitive-radar.log`：竞品采购雷达完整日志；
- `run_summary.json`：运行元数据、指标、诊断和两个业务 Summary；
- `run_summary.md`：Actions Summary 和人工排查入口。

运行数据不会提交进 Git 历史，而是通过 Actions Artifact 保存，避免仓库持续膨胀。

## 4. 本地运行

完整运行：

```bash
cd 自动化情报收集系统
python3 scripts/run_daily_ops.py --date today
```

指定日期：

```bash
python3 scripts/run_daily_ops.py --date 2026-07-23
```

仅用于排查、不让失败返回非零状态：

```bash
python3 scripts/run_daily_ops.py --date today --allow-failure
```

跳过测试：

```bash
python3 scripts/run_daily_ops.py --date today --skip-tests
```

macOS `launchd` 仍调用：

```bash
scripts/run_daily.sh
```

该脚本已经统一转到运维执行器。

## 5. 状态判定规则

### 5.1 Failed

满足任一条件：

- 单元测试退出码非 0；
- 通用 Pipeline 退出码非 0；
- 竞品雷达退出码非 0；
- 应生成的业务 Summary 缺失；
- 已配置 Notion，但同步结果为失败。

### 5.2 Warning

步骤均执行成功，但出现以下情况：

- 通用情报原始采集为 0；
- 竞品信号为 0；
- 10 个重点竞品覆盖率低于 30%。

### 5.3 Success

步骤全部成功，Summary 完整，且未触发错误或质量警告。

## 6. 自动诊断代码

| 诊断代码 | 含义 | 优先检查 |
| --- | --- | --- |
| `STEP_FAILED_UNIT_TESTS` | 单元测试失败 | `01_unit-tests.log` |
| `STEP_FAILED_GENERAL_PIPELINE` | 通用 Pipeline 失败 | `02_general-pipeline.log` |
| `STEP_FAILED_COMPETITIVE_RADAR` | 竞品雷达失败 | `03_competitive-radar.log` |
| `GENERAL_SUMMARY_MISSING` | 通用 Summary 未生成 | Pipeline 异常、目录权限 |
| `COMPETITIVE_SUMMARY_MISSING` | 竞品 Summary 未生成 | 竞品脚本异常、配置错误 |
| `GENERAL_ZERO_RAW_ITEMS` | 通用采集为 0 | RSSHub、网络、数据源配置 |
| `COMPETITIVE_ZERO_SIGNALS` | 无有效竞品信号 | 数据源、关键词、过滤规则 |
| `LOW_COMPETITOR_COVERAGE` | 竞品覆盖率低于 30% | 零信号竞品的独立源 |
| `GENERAL_NOTION_FAILED` | 通用 Notion 失败 | Token、数据库/页面 ID、授权 |
| `COMPETITIVE_NOTION_FAILED` | 竞品 Notion 失败 | 竞品父页面 ID、授权 |

## 7. 标准排查流程

### 第一步：判断失败范围

查看 Actions Summary：

- 只有测试失败：优先处理代码或测试；
- 通用 Pipeline 失败：查看采集、规则、报告和 Notion；
- 竞品雷达失败：查看竞品配置、RSS 和预算/归属规则；
- 全部失败：优先检查 Python 环境、仓库结构和依赖文件。

### 第二步：下载 Artifact

打开对应 Run，下载 `ai-intelligence-<run_id>`。

先查看：

```text
data/run_logs/latest_run.md
```

再根据失败步骤进入对应 `.log`。

### 第三步：确认是否为配置问题

检查 GitHub：

```text
Settings -> Secrets and variables -> Actions
```

重点 Secret：

- `NOTION_TOKEN`；
- `NOTION_DATABASE_ID`；
- `NOTION_PARENT_PAGE_ID`；
- `NOTION_COMPETITIVE_PARENT_PAGE_ID`；
- `RSSHUB_BASE_URL`。

不要把 Secret 内容复制到 Issue、PR、日志或排查记录中。

### 第四步：手动重跑

进入失败的 Actions Run：

```text
Re-run jobs
```

也可以从工作流首页选择：

```text
Run workflow
```

手动触发时，`trigger` 会记录为 `workflow_dispatch`。

### 第五步：记录问题

确认根因后，在《故障排查记录》中新增一条记录，至少包含：

- 发生时间和 Run ID；
- 影响范围；
- 诊断代码；
- 根因；
- 临时处理；
- 永久修复；
- 验证结果。

## 8. 高频问题处理

### 8.1 通用采集为 0

检查顺序：

1. `RSSHUB_BASE_URL` 是否可访问；
2. 公共 RSSHub 是否限流；
3. `config/sources.json` 中源地址是否失效；
4. 是否因为 `--skip-low-stability` 排除了过多来源；
5. 网络或证书错误是否集中出现。

### 8.2 竞品覆盖率过低

检查：

1. `config/competitor_watchlist.json` 中 `news_feed_url`；
2. 竞品别名是否覆盖新产品名称；
3. 官方域名是否变更；
4. Google News RSS 是否没有索引该站点；
5. 关键词是否过窄。

覆盖率低不等于系统失败，但连续 3 天低于 30% 应进入数据源维护。

### 8.3 Notion 同步失败

检查：

1. Token 是否过期或写错；
2. Integration 是否被授权到目标页面和数据库；
3. 页面 ID 和数据库 ID 是否混用；
4. 数据库字段是否与程序要求一致；
5. Notion API 是否限流或临时故障。

Notion 失败不应删除本地日报。日报和 Summary 仍应保留在 Artifact。

### 8.4 工作流没有运行

检查：

1. 工作流文件是否已在默认分支 `main`；
2. Actions 是否被仓库禁用；
3. 定时任务是否刚合并并错过当天 08:30；
4. 是否存在仍在运行的同组任务；
5. 先通过 `workflow_dispatch` 手动验证。

## 9. 故障等级

| 等级 | 定义 | 示例 | 处理要求 |
| --- | --- | --- | --- |
| S1 | 系统连续无法运行或数据丢失 | 连续多日无日报且无日志 | 立即处理 |
| S2 | 核心模块失败 | 通用或竞品 Pipeline 持续失败 | 当日处理 |
| S3 | 部分能力异常 | Notion 失败、部分源失效 | 3 个工作日内处理 |
| S4 | 数据质量提示 | 竞品覆盖率偏低、偶发零信号 | 周复盘处理 |

## 10. 安全与保留策略

- 日志会对 `NOTION_TOKEN`、`OPENAI_API_KEY` 和 `LLM_API_KEY` 的实际值进行替换脱敏；
- 不在日志中主动打印完整 Secret；
- Artifact 默认保留 30 天；
- 长期结论写入《故障排查记录》，不长期保存全部原始日志；
- 涉及客户、合同、个人数据时，应进一步增加字段级脱敏。
