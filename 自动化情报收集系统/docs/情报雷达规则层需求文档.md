# AI Agent 情报雷达规则层需求文档

## 1. 背景

当前项目已经有 `config/sources.json`，用于定义 AI/Agent 情报系统从哪里采集信息。仅有信息源还不够，因为系统会面临三个问题：

1. 抓到的信息越来越多，但不知道哪条最重要；
2. 同一事件会被多个来源重复报道，需要归并；
3. 通用 AI 新闻不一定对唐文怡的求职、产品复盘、商业化判断有价值。

因此需要新增一层“规则配置”，把原始信息从“新闻列表”升级为“可判断、可排序、可输出、可复盘”的情报系统。

## 2. 目标

规则层的目标是让系统回答 5 个问题：

1. 这条信息是什么类型？
2. 这条信息可信不可信？
3. 这条信息重要不重要？
4. 这条信息和我有什么关系？
5. 这条信息应该进入日报、知识库、面试案例，还是忽略？

## 3. 新增配置文件说明

### 3.1 `config/scoring_rules.json`

#### 定义

评分规则文件，负责判断每条情报的优先级。

#### 解决的问题

没有评分规则时，系统只能按时间排序；有评分规则后，系统可以按价值排序。

#### 核心字段

| 字段 | 含义 | 是否可扩展 |
|---|---|---|
| `score_range` | 1-5 分的含义 | 可扩展，但建议保持 1-5 分 |
| `final_score_formula` | 总分公式和权重 | 可调整权重 |
| `dimensions` | 各评分维度 | 可新增维度 |
| `priority_thresholds` | 日报/周报/归档阈值 | 可按信息量调整 |
| `action_mapping` | 分数到动作的映射 | 可新增动作 |

#### 当前评分维度

| 维度 | 作用 |
|---|---|
| `freshness_score` | 判断信息新不新 |
| `evidence_score` | 判断来源可信度 |
| `trend_score` | 判断是否代表趋势 |
| `business_score` | 判断商业价值 |
| `personal_relevance_score` | 判断和唐文怡当前目标的关系 |
| `risk_score` | 判断风险/预警价值 |

#### 总分公式

```text
final_score = freshness*0.15 + evidence*0.20 + trend*0.20 + business*0.20 + personal*0.20 + risk*0.05
```

#### 使用示例

如果某条信息是“豆包智能体能力下架”：

| 维度 | 可能分数 | 原因 |
|---|---:|---|
| freshness_score | 5 | 当天发生 |
| evidence_score | 4 | 如果来自官方公告或可信媒体 |
| trend_score | 5 | 影响国内 Agent 平台方向 |
| business_score | 5 | 影响普通 Agent 公司商业化 |
| personal_relevance_score | 5 | 可转成面试表达和产品判断 |
| risk_score | 5 | 涉及下架/监管/内容安全 |

系统应把它放进日报 Top 5，并标记为 `track_follow_up`、`convert_to_interview_story`、`create_product_insight`。

#### 扩展规则

新增评分维度时，必须满足：

1. 有明确的 1-5 分规则；
2. 能被程序或 LLM 判断；
3. 在 `final_score_formula` 中配置权重；
4. 所有权重总和建议等于 1。

### 3.2 `config/event_types.json`

#### 定义

事件类型字典，负责把情报归类。

#### 解决的问题

同样是 AI 新闻，可能是政策、产品、融资、招聘、下架、开源版本等。事件类型可以让日报结构稳定，也方便后续统计趋势。

#### 核心字段

| 字段 | 含义 | 是否可扩展 |
|---|---|---|
| `event_types` | 事件类型列表 | 可新增 |
| `classification_rules` | 多标签和主标签规则 | 可调整 |
| `priority_order` | 主标签优先级 | 可调整 |

#### 当前事件类型

| id | 中文名 | 典型场景 |
|---|---|---|
| `policy_regulation` | 政策 / 监管变化 | 网信办、工信部、EU AI Act |
| `standard_update` | 标准 / 规范变化 | AI 国标、NIST 框架 |
| `product_launch` | 新产品 / 新平台发布 | 新 Agent 平台、新 API |
| `feature_update` | 功能更新 / Changelog | 工作流、工具调用、知识库更新 |
| `pricing_change` | 价格 / 计费 / 限流变化 | API 定价、额度、限流 |
| `shutdown_or_removal` | 下架 / 关闭 / 整改 | 智能体下架、功能停服 |
| `open_source_release` | 开源版本发布 | Dify、LangGraph、vLLM Release |
| `security_safety_risk` | 安全 / 内容安全 / 风险事件 | 提示词注入、拟人化、未成年人 |
| `funding_commercialization` | 融资 / 商业化信号 | 融资、收入、客户、收购 |
| `hiring_signal` | 招聘 / 组织信号 | AI Agent PM、AI 合规岗 |
| `market_opinion` | 市场观点 / 早鸟信号 | X、即刻、小红书讨论 |
| `case_study` | 案例 / 落地实践 | 企业知识库、客服、销售 Agent 案例 |
| `other` | 其他 | 无法判断的信息 |

#### 多标签规则

一条信息可以有多个标签。

例如：

```text
豆包智能体下架 = shutdown_or_removal + policy_regulation + security_safety_risk
```

主标签按照 `priority_order` 选择最高优先级。

#### 扩展规则

新增事件类型时，必须填写：

1. `id`
2. `name`
3. `description`
4. `keywords`
5. `default_priority`
6. `recommended_actions`

如果只是关键词变化，不要新增类型，优先扩展已有类型的 `keywords`。

### 3.3 `config/output_schema.json`

#### 定义

输出结构文件，负责定义单条情报、日报、周报、知识库条目的字段。

#### 解决的问题

没有输出结构时，日报容易每次长得不一样，后续也很难同步到 Notion、飞书、数据库。

#### 核心结构

| 模块 | 作用 |
|---|---|
| `single_item_schema` | 单条情报标准字段 |
| `daily_report_schema` | 日报栏目结构 |
| `weekly_report_schema` | 周报栏目结构 |
| `markdown_templates` | Markdown 输出模板 |
| `export_targets` | 导出目标 |

#### 单条情报关键字段

| 字段 | 作用 |
|---|---|
| `item_id` | 唯一 ID，避免重复 |
| `title` | 情报标题 |
| `source_name` | 来源 |
| `source_url` | 原文链接 |
| `channel` | 情报频道 |
| `event_types` | 多事件标签 |
| `primary_event_type` | 主事件标签 |
| `clean_summary` | 事实摘要 |
| `why_it_matters` | 为什么重要 |
| `product_insight` | 产品启发 |
| `career_insight` | 求职/面试启发 |
| `risk_insight` | 风险判断 |
| `scores` | 评分结果 |
| `recommended_actions` | 建议动作 |
| `status` | 处理状态 |

#### 日报栏目

1. 今日最高优先级信号；
2. 政策 / 合规 / 风险；
3. Agent 产品 / 平台更新；
4. Infra / API / 算力；
5. 投融资 / 商业化；
6. 招聘 / 岗位 / 能力信号；
7. 今日结论。

#### 设计原则

事实和判断要分开：

| 字段 | 内容 |
|---|---|
| `clean_summary` | 只写事实 |
| `why_it_matters` | 写重要性判断 |
| `product_insight` | 写产品启发 |
| `career_insight` | 写求职/面试启发 |

这样可以避免把观点伪装成事实。

### 3.4 `config/personal_focus.json`

#### 定义

个人关注规则文件，负责判断一条通用 AI 信息对唐文怡有没有价值。

#### 解决的问题

AI 信息很多，但不是每条都和当前目标有关。这个文件用于把通用情报转成个人可用的职业判断。

#### 当前个人定位

```text
AI / ToB 产品经理，关注 Agent、企业知识库、AI 工作流、商业化和合规。
```

#### 当前高权重主题

| 主题 | 权重 | 价值 |
|---|---:|---|
| AI Agent / 智能体 | 5 | 当前主线 |
| 企业知识库 / RAG | 5 | 与过去小智、知识库经验相关 |
| 工作流自动化 | 5 | Agent 产品落地核心能力 |
| AI 合规 / 内容安全 | 5 | 国内 Agent 风险边界 |
| 商业化产品经理 | 5 | 求职和岗位判断核心 |
| API / 模型网关 / 中转 | 4 | 与 AI Infra、平台商业化相关 |
| 岗位 / 能力信号 | 4 | 判断市场需要什么产品经理 |
| 开源 Agent 技术栈 | 3 | 辅助理解产品能力，不做底层技术深挖 |

#### 高价值问题

每条高价值信息最好能回答下面至少一个问题：

1. 是否影响 Agent 产品商业化路径？
2. 是否说明平台规则或监管边界变化？
3. 是否能转成面试中的产品判断案例？
4. 是否说明某类 AI 产品正在被市场验证？
5. 是否影响 AI 产品经理需要具备的能力？
6. 是否暴露普通 Agent 公司会遇到的风险？
7. 是否能沉淀成知识库标签或选题？

#### 使用场景

| 场景 | 输出 |
|---|---|
| 面试表达 | 背景-外部变化-产品判断-我会怎么做 |
| 产品复盘 | 产品能力-目标用户-商业价值-风险边界 |
| 机会清单 | 客户是谁-痛点是什么-为什么现在可做-最小验证方式 |

## 4. 规则层整体流程

```text
sources.json
  ↓
采集原始信息
  ↓
event_types.json：判断事件类型
  ↓
scoring_rules.json：计算情报分数
  ↓
personal_focus.json：判断对唐文怡的个人价值
  ↓
output_schema.json：生成单条情报 / 日报 / 周报 / 知识库条目
```

## 5. 单条情报处理流程

### Step 1：采集

从 `sources.json` 获取来源、频道、关键词、RSS 或网页链接。

### Step 2：清洗

去除广告、导航、重复标题、无效正文。

### Step 3：去重归并

同一事件多来源报道时，保留官方源或最早可信源，其他源进入 `related_items`。

### Step 4：事件分类

使用 `event_types.json` 判断事件类型。

允许多标签。

### Step 5：评分

使用 `scoring_rules.json` 计算：

```text
freshness_score
evidence_score
trend_score
business_score
personal_relevance_score
risk_score
final_score
```

### Step 6：个人价值判断

使用 `personal_focus.json` 判断是否能转化为：

1. 面试案例；
2. 产品复盘；
3. 机会清单；
4. 知识库条目；
5. 仅归档或忽略。

### Step 7：输出

使用 `output_schema.json` 生成 Markdown、JSONL、Notion 或飞书文档。

## 7. 可扩展设计

### 7.1 新增信息源

只改 `sources.json`。

必须包含：

```json
{
  "name": "来源名称",
  "channel": "频道",
  "evidence_level": "证据等级",
  "source_type": "rss/web_page/rsshub_github_release/rsshub_wechat",
  "priority": 1,
  "language": "zh/en/mixed",
  "stability": "high/medium/low",
  "url": "官网链接",
  "monitor_keywords": [],
  "feed_url": "可选，原生 RSS",
  "rsshub_path": "可选，RSSHub 路径"
}
```

### 7.2 新增事件类型

只改 `event_types.json`。

不要为了一个新闻热点随便加类型。只有当某类事件会长期出现，才新增事件类型。

### 7.3 调整评分权重

只改 `scoring_rules.json`。

场景建议：

| 当前目标 | 建议提高权重 |
|---|---|
| 求职 | `personal_relevance_score` |
| 创业机会 | `business_score` |
| 合规研究 | `risk_score` 和 `evidence_score` |
| 行业趋势 | `trend_score` |
| 快讯日报 | `freshness_score` |

### 7.4 调整个人关注方向

只改 `personal_focus.json`。

例如未来你不再关注求职，而是关注创业，可以新增：

```json
{
  "id": "startup_opportunity",
  "name": "创业机会",
  "weight": 5,
  "keywords": ["痛点", "付费", "中小企业", "自动化", "低成本", "MVP"]
}
```

### 7.5 新增输出渠道

只改 `output_schema.json`。

比如新增飞书、Notion、邮件、GitHub Issue。

## 8. 验收标准

### 配置层验收

- 所有 JSON 文件格式合法；
- 所有配置都有 version；
- 所有评分规则都有 1-5 分定义；
- 所有事件类型都有 id、name、description、keywords；
- 输出结构能覆盖单条情报、日报、周报。

### 功能层验收

输入一条情报：

```text
OpenAI 发布 Agents SDK 新版本，增强 tracing 和 handoff 能力。
```

系统应输出：

```json
{
  "event_types": ["feature_update", "open_source_release"],
  "primary_event_type": "feature_update",
  "scores": {
    "freshness_score": 5,
    "evidence_score": 4,
    "trend_score": 4,
    "business_score": 4,
    "personal_relevance_score": 5,
    "risk_score": 1,
    "final_score": 4.05
  },
  "recommended_actions": ["save_to_knowledge_base", "create_product_insight", "convert_to_interview_story"]
}
```

输入一条情报：

```text
某社媒用户爆料某智能体产品即将下架。
```

系统应输出：

```json
{
  "event_types": ["market_opinion", "shutdown_or_removal"],
  "primary_event_type": "shutdown_or_removal",
  "evidence_score": 1,
  "risk_score": 5,
  "recommended_actions": ["track_follow_up"],
  "note": "不能作为事实结论，需等待官方或可信媒体确认"
}
```

## 9. 非目标

当前阶段不做：

1. 自动发社媒；
2. 完全自动判断事实真伪；
3. 高频爬取需要登录的网站；
4. 用社媒爆料直接生成结论；
5. 大而全的 AI 新闻门户。

本项目的定位不是资讯站，而是唐文怡个人使用的 AI/Agent 产品情报雷达。

## 10. 下一步建议

建议下一步让 Codex 实现：

1. `src/config_loader.py`：加载所有配置；
2. `src/rule_engine.py`：事件分类和评分；
3. `src/report_generator.py`：生成 Markdown 日报；
4. `tests/test_rule_engine.py`：用样例验证规则。

优先实现规则引擎，不要一开始就做复杂爬虫。先证明：输入一条信息，系统能判断它是否重要。
